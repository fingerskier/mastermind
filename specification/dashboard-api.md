# Dashboard API

Status: Phase 1 MVP contract.

The local dashboard is served by `landsraad dashboard` from a Fastify server scoped to exactly one council root. The server reads council files directly and does not create a hosted backend or external state store.

## CLI

```text
landsraad dashboard
landsraad --council <path> dashboard
landsraad --council <path> dashboard --host 127.0.0.1 --port 4173
npm start
npm run dev
```

The command prints the local URL and keeps the process running until interrupted.

From the package root, `npm start` runs the built dashboard against the dogfood council by default. `npm run dev` starts the Fastify dashboard server and Vite dev server together; the Vite UI proxies `/api` requests to the dashboard server so React hot reload can run without changing API paths. `LANDSRAAD_COUNCIL`, `LANDSRAAD_DASHBOARD_HOST`, `LANDSRAAD_DASHBOARD_PORT`, `LANDSRAAD_UI_PORT`, and `LANDSRAAD_API_TARGET` can override the local development defaults.

## API Endpoints

All paths are relative to the dashboard server origin.

- `GET /api/health`: server health and scoped council root.
- `GET /api/overview`: combined council, Secretary, agents, jobs, projects, memory, run summaries, and counts.
- `GET /api/council`: council config and canonical source directories.
- `GET /api/secretary`: Secretary definition and persona text.
- `GET /api/agents`: Secretary and councillor definitions with persona previews.
- `GET /api/jobs`: job definitions with brief, status text, schedule, and recent runs.
- `GET /api/projects`: project directories with brief, notes, decisions, and file summaries.
- `GET /api/memory`: shared memory index, facts JSON, and memory file summaries.
- `GET /api/runs`: recent run summaries across jobs.
- `GET /api/runs/:jobId/:runId`: run detail with `run.json`, `input.md`, `transcript.md`, `output.md`, and `events.jsonl`.
- `GET /api/retrieval/search?query=<text>&limit=<n>&mode=hybrid`: retrieval search over council files.
- `POST /api/dashboard-objects/validate`: validate structured dashboard objects and return either a supported renderer or JSON fallback.

## Run Detail

The run detail response preserves the plain-file model. File contents are returned under stable keys:

```json
{
  "files": {
    "runJson": "{...}",
    "inputMd": "# Task Packets",
    "transcriptMd": "# Transcript",
    "outputMd": "# Run Output",
    "eventsJsonl": "{\"kind\":\"process-started\"}\n"
  }
}
```

`eventsJsonl` is also parsed into an `events` array when possible. Invalid JSONL lines are returned as parse-error entries instead of failing the whole run detail view.

## Structured Objects

The dashboard validates structured objects from `run.json` fields named `structuredObjects` or `dashboardObjects`, and from JSON code fences in `output.md`. Valid forms, tables, charts, and status cards get a type-specific renderer. Invalid or unsupported objects remain visible as formatted JSON with validation warnings.

## Boundaries

- API reads are council-root scoped.
- The Phase 1 dashboard is inspection-first; job execution and review actions remain CLI/file workflows.
- Retrieval search uses the existing local rebuildable index.
- The bundled UI is generated by Vite/React and served as static files by the CLI server.
- Local development serves the UI through Vite while proxying `/api` to the Fastify dashboard server.
